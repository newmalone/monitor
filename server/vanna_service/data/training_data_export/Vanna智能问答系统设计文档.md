## 1. 项目概述

### 1.1 项目背景

运维管理系统需要一种自然语言方式查询设备数据，降低运维人员使用 SQL 的门槛。Vanna AI 智能问答系统基于 Text-to-SQL 技术，将自然语言问题自动转换为 SQL 查询，实现"问即所得"的数据查询体验。

### 1.2 项目目标

| 目标 | 指标 | 说明 |
|------|------|------|
| 响应速度 | 已训练问题 <0.3 秒 | 关键词匹配快速路径 |
| 回答准确率 | 基准测试 ≥90% | 训练数据覆盖核心业务场景 |
| 系统可用性 | 99%+ | 三层容错（关键词匹配 → LLM → RAG兜底） |
| 可扩展性 | 支持多LLM | 火山引擎/DeepSeek/OpenAI 可切换 |

### 1.3 技术选型

| 层级 | 技术 | 选型理由 |
|------|------|----------|
| 前端 | React + TypeScript + Ant Design | 企业级 UI 组件库，支持图表可视化 |
| 后端代理 | Node.js + Express | 轻量级代理层，处理文件上传和数据双写 |
| AI 引擎 | Python + FastAPI + Vanna | Text-to-SQL 核心框架，支持训练和推理 |
| LLM | 火山引擎 ARK (doubao-seed-2-1-pro) | 国产大模型，中文理解能力强 |
| 数据库 | SQLite | 轻量级，无需额外部署，适合中小规模 |
| 向量存储 | 自研 SQLite 向量存储 | 避免 ChromaDB 依赖，确定性 MD5 嵌入 |

### 2.2 服务职责划分

| 服务 | 端口 | 核心职责 | 不做什么 |
|------|------|----------|----------|
| 前端 (React) | 3000 | UI 渲染、用户交互、图表展示 | 不直接调用 Python API |
| 后端代理 (Node.js) | 3001 | API 代理、文件上传、数据双写 | 不做 AI 推理 |
| Vanna AI (Python) | 3002 | Text-to-SQL、训练管理、对话管理 | 不处理文件上传 |

### 2.3 通信协议

```
前端 ←→ 后端代理：HTTP/JSON (RESTful)
后端代理 ←→ Vanna AI：HTTP/JSON + SSE (流式)
Vanna AI ←→ LLM API：HTTPS/OpenAI 兼容协议
Vanna AI ←→ SQLite：本地文件 I/O
```

## 5. 核心模块设计

### 5.1 VannaManager - 核心管理器

**职责**：协调 SQL 生成、执行、回答生成的完整流程。

**核心方法**：

| 方法 | 职责 | 输入 | 输出 |
|------|------|------|------|
| `ask()` | 完整问答流程 | question, context | {answer, sql, data, validation} |
| `ask_stream()` | 流式问答 | question, context | Generator[(event_type, data)] |
| `_generate_sql()` | SQL 生成（三层策略） | question, context, last_error | sql_str or None |
| `_match_sql_by_keyword()` | 关键词匹配 | question | sql_str or None |
| `_safe_run_sql()` | 安全执行 SQL | sql_str | list[dict] or {error} |
| `_normalize_sql_date()` | 动态日期替换 | sql_str | sql_str (日期已替换) |
| `_generate_answer()` | 生成自然语言回答 | context dict | answer_str |
| `_append_rag_suggestions()` | RAG 推荐追加 | result, question | (修改 result) |
| `train_sql()` | 训练 SQL 问答对 | question, sql | id_str |
| `train_ddl()` | 训练 DDL | ddl_str | id_str |
| `train_doc()` | 训练文档 | content | id_str |
| `benchmark_test()` | 基准测试 | questions | {success_rate, results} |

**关键设计决策**：

1. **三层 SQL 生成策略**：精确匹配 → 意图匹配 → 关键词评分 → LLM
2. **SQL 执行重试机制**：最多 2 次重试，连续相同错误时停止
3. **动态日期替换**：训练数据中的硬编码日期自动替换为最新快照日期
4. **结果验证**：empty/single/multi/anomaly 四类状态
5. **RAG 兜底**：失败/空结果时从训练库检索相似问题推荐

### 5.2 ContextManager - 对话上下文管理

**职责**：管理多轮对话历史、实体提取、追问意图分析。

**实体识别模式**：

| 实体类型 | 正则模式 | 示例 |
|----------|----------|------|
| date | `\d{4}[-/]\d{1,2}[-/]\d{1,2}\|今天\|昨天\|最近\d+天` | "2026-06-16", "最近7天" |
| region | `锡山区\|惠山区\|滨湖区\|梁溪区\|新吴区\|经开区` | "锡山区", "经开区" |
| manufacturer | `海康威视\|大华\|宇视\|华为\|中兴` | "海康威视", "大华" |
| device_type | `MEC\|OLT\|ONU\|交换机\|路由器\|摄像头` | "摄像头", "交换机" |

**追问意图类型**：

| 意图类型 | 触发关键词 | 说明 |
|----------|-----------|------|
| diagnosis | 为什么、原因、怎么回事、故障 | 诊断类追问 |
| compare | 对比、比较、差异、跟、比 | 对比类追问 |
| drill_down | 具体有哪些、详情、详细、列出 | 下钻类追问 |
| refine_filter | 呢、那、还有、换成、改为 | 过滤细化追问 |
| new_query | (默认) | 全新问题 |

### 5.3 AnswerGenerator - 回答生成器

**职责**：基于 SQL、数据、验证结果生成自然语言回答。

**降级策略**：

```
LLM 可用 → 调用 LLM 生成自然语言回答
    ↓ (失败)
LLM 不可用 → 格式化输出（列出数据行）
    ↓ (无数据)
空结果 → 提示用户放宽条件
    ↓ (执行失败)
失败场景 → 提示用户换种方式提问 + RAG 推荐
```

### 5.4 ResultValidator - 结果验证器

**职责**：对 SQL 执行结果进行分类验证和异常检测。

**验证状态**：

| 状态 | 条件 | 说明 |
|------|------|------|
| empty | 无数据或 `[{}]` | 查询无结果 |
| single | 1 行数据 | 单值结果（如总数） |
| multi | 多行数据，无异常 | 正常多行结果 |
| anomaly | 多行数据，检测到异常值 | 数据存在异常 |

**异常检测规则**：
1. 任一数值偏离中位数 3 倍以上
2. 变异系数（std/mean）> 1

### 5.5 SQLiteVectorStore - 向量存储

**职责**：存储和检索训练数据的向量表示。

**关键设计**：

- **嵌入生成**：使用 MD5 确定性哈希生成 1536 维向量（避免外部 API 依赖）
- **检索方式**：由于嵌入是确定性的，不依赖余弦相似度，而是按类型全量返回
- **存储格式**：SQLite 单表，包含 id、document、embedding、metadata

## 8. 性能设计

### 8.1 响应时间目标

| 场景 | 目标 | 实际 | 实现方式 |
|------|------|------|----------|
| 已训练问题 | <0.3 秒 | ~0.014 秒 | 关键词匹配（MD5向量，无需LLM） |
| 未训练问题 | 3-10 秒 | 3-8 秒 | LLM 调用（火山引擎 ARK） |
| 流式首字 | <1 秒 | ~0.5 秒 | SSE 逐字推送 |
| 对话列表加载 | <0.5 秒 | ~0.1 秒 | SQLite 索引查询 |

### 8.2 性能优化策略

**1. 关键词匹配三层策略**
- 策略0：精确匹配（O(n)，n=训练对数量）
- 策略1：意图优先匹配（O(n*m)，m=意图规则数量）
- 策略2：关键词评分（O(n*k)，k=关键词数量）
- 策略3：LLM 生成（仅在以上全部失败时调用）

**2. 确定性 MD5 嵌入**
- 避免外部 embedding API 调用
- 相同文本始终生成相同向量
- 训练时一次性生成，查询时直接检索

**3. 动态日期替换**
- SQL 执行前自动替换硬编码日期
- 避免训练数据过期问题

**4. SQL 执行重试**
- 最多 2 次重试
- 连续相同错误时立即停止

### 8.3 容量规划

| 指标 | 当前值 | 建议上限 | 说明 |
|------|--------|----------|------|
| 训练 SQL 对 | 240+ | 500 | 过多会降低匹配精度 |
| 训练 DDL | 6 | 20 | 表数量有限 |
| 训练文档 | 10 | 50 | 文档过多会增加 LLM 上下文 |
| 对话记录 | 60+ | 1000 | SQLite 可轻松支持 |
| 设备数据 | 17,871 | 100,000 | SQLite 单表建议 <100万行 |

## 9. 安全设计

### 9.1 SQL 注入防护

```python
# 只允许 SELECT 和 WITH 语句
sql_upper = sql.strip().upper()
if not sql_upper.startswith("SELECT") and not sql_upper.startswith("WITH"):
    return {"_error": "只允许 SELECT/WITH 语句", "_sql": sql}
```

### 9.2 数据验证

- **ResultValidator**：验证 SQL 执行结果类型和异常值
- **输入过滤**：用户问题中的特殊字符在 LLM 调用前清理
- **输出清理**：LLM 返回的 SQL 去除 Markdown 标记和多余说明

### 9.3 API 安全

- **CORS 配置**：当前允许所有来源（生产环境应配置具体域名）
- **认证**：通过 user_id 参数区分用户（生产环境应接入认证系统）
- **速率限制**：当前未实现（生产环境建议添加）
